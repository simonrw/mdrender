## Context

`mdr` currently behaves like a Markdown-aware `cat`: it reads a file or stdin, renders Markdown when stdout is a terminal, and writes either rendered terminal output or raw Markdown directly to stdout. That keeps pipelines safe, but longer rendered documents still fill terminal scrollback, which makes reading awkward.

The first implementation deliberately excluded interactive paging. This change revisits that boundary in the narrowest useful way: keep the renderer stateless and pipe-friendly, but send rendered interactive output to `less -R` by default.

## Goals / Non-Goals

**Goals:**

- Page rendered TTY output through `less -R` by default.
- Preserve the current raw passthrough behavior for redirected and piped stdout in `--render auto`.
- Add an explicit `--pager auto|always|never` control.
- Keep ANSI styling visible in the default pager by using `less -R`.
- Keep tests deterministic by isolating pager selection and process spawning behind a small boundary.

**Non-Goals:**

- Do not respect `$PAGER` in this change.
- Do not add a full TUI, alternate application renderer, scroll state, search UI, or keyboard event loop.
- Do not use `bat`, `mdcat`, or other Markdown-aware pagers that could re-render output.
- Do not change Markdown parsing, layout, wrapping, highlighting, or raw passthrough semantics.

## Decisions

### Use `less -R` as the only pager command

Hard-code the pager command as `less -R`. The `-R` flag allows ANSI color and style sequences generated by `mdr` to pass through safely, while less still manages the terminal screen and scrolling.

Alternatives considered:

- Respect `$PAGER`: convenient, but custom values such as `bat` can re-render or reinterpret already-rendered Markdown.
- Use `less` without `-R`: simpler, but colored output would display raw escape sequences or lose useful styling.
- Add an embedded pager: more control, but it drags the project toward a TUI and duplicates mature terminal behavior.

### Add an explicit pager mode

Add a `PagerMode` enum with `auto`, `always`, and `never`, exposed as `--pager`.

`auto` is the default. When stdout is a TTY and the command is rendering, `mdr` sends rendered output to `less -R`. When stdout is not a TTY in `--render auto`, `mdr` keeps writing raw Markdown directly to stdout and never launches a pager.

`never` disables pager use and writes rendered output directly to stdout whenever rendering is selected.

`always` requests pager use for rendered output. If pager use is impossible or `less` cannot be launched, the command fails with a clear error rather than silently changing output mode.

### Keep paging at the CLI output boundary

Do not push pager behavior into the parser, layout, renderer, or ANSI emitter. The core render path should continue to produce a complete string from Markdown source and render options. The CLI decides whether that string is written to stdout or to the stdin of a pager process.

This keeps existing render tests stable and makes pager behavior testable as mode selection plus process invocation.

### Treat pager failures as user-visible command errors

If `less` is missing, cannot be spawned, or exits unsuccessfully, return an error from the CLI. Rendering has already succeeded at that point, but the requested presentation failed, so the user should get a clear stderr message and a non-zero exit.

## Risks / Trade-offs

- `less` may not exist on every platform -> Surface a clear error and leave `--pager never` as an escape hatch for direct output.
- Interactive pager tests can be brittle -> Keep process spawning behind a small helper so unit tests can verify decisions without requiring an actual interactive terminal.
- Default behavior changes for TTY users -> Provide `--pager never` and document it prominently.
- Paging a stdin stream still requires reading the full input before display -> This matches the existing non-streaming renderer and avoids changing layout architecture.
